01

物理实现：数据到底走哪根线？

NCCL 的核心工作是：探测硬件拓扑 → 把 GPU 排成一个或多个"通信环 / 树" → 在最快的链路上搬数据。先看一台典型的 8 卡服务器长什么样。

▣ 一台 8×GPU 节点的连线图 节点内 = NVLink/NVSwitch（快） · 节点间 = NIC→InfiniBand（慢一个量级）
NVLink / NVSwitch · 节点内最快路径
PCIe · 慢，回退路径
NIC → InfiniBand/RoCE · 跨节点

三层链路速度差，是理解一切的前提

下表给的是链路的物理峰值带宽(硬件标称的单向/聚合上限)。注意：你实测一次 all-reduce 跑出的"算法带宽"会明显低于这个数——因为集合操作要在环上走 2(N−1)/N 倍的数据、还有启动延迟。两个口径都对，但别混。

链路物理峰值带宽用在哪记忆点
NVLink / NVSwitch~600–900 GB/s同节点 GPU↔GPU快到像本地内存
PCIe Gen5~64 GB/sGPU↔CPU，无 NVLink 时的回退慢一个数量级
InfiniBand / RoCE~25–50 GB/s/卡跨节点（配 GPUDirect RDMA）跨机就靠它，绕不开
物理带宽 ≠ 算法带宽，这是新手最常踩的坑。 上表是链路的物理峰值（硬件标称上限）。但你用测速工具（如 nccl-tests）实测一次 all-reduce，量出来的是算法带宽 (algo bandwidth)——明显更低：PCIe ~5–15 GB/s、NVLink ~50–120 GB/s、NVSwitch ~150–250 GB/s。为什么低这么多？因为一次 all-reduce 不是"把数据搬一遍"就完事：在环上，每份数据要被搬运 2(N−1)/N ≈ 2 倍的量（先 ReduceScatter 再 AllGather，见下一节），再叠加每一步的启动延迟。所以"实测约为物理上限的一半上下"是正常现象，不是你配置错了。
STEP 1 · BOOTSTRAP

拓扑探测

启动时 NCCL 扫描每张卡的 NVLink、PCIe 路径、NIC，建出一张硬件图。NCCL_DEBUG=INFO 能打印它最终选了什么路径。

STEP 2 · ALGORITHM

搭环 / 搭树

把 GPU 排成一个或多个逻辑环（Ring）或树（Tree）。环带宽最优，树延迟最优，NCCL 按规模自动选。

STEP 3 · TRANSPORT

零拷贝搬运

GPU↔GPU 直接走 NVLink（P2P），跨机走 GPUDirect RDMA，绕过 CPU。这就是 NCCL 比"自己写 MPI"快的原因。

为什么是"环"？—— 带宽最优的朴素直觉

朴素做法：所有卡把数据发给 0 号卡求和，再发回去 → 0 号卡的网卡被打爆，成为瓶颈。环的聪明之处：每张卡只和左右邻居通信，所有链路同时满载，没有任何一条线是热点。下一节的交互动画会把这一点演出来。

但 NCCL ≠ 只有环：它有一整套算法，按规模自动选

"环带宽最优"只在中等规模成立。卡一多，环的步数 2(N−1) 线性增长，延迟就吃不消了。NCCL 内部其实有多套算法，启动时按 GPU 数、消息大小、拓扑自动挑选（NCCL_DEBUG=INFO 会打印 Ring / Tree / NVLS 等字样）：

算法结构强在典型触发
RingGPU 排成环，邻居传递带宽最优、无热点大消息、中等规模
Treedouble binary tree（双二叉树）延迟随卡数 对数级增长多节点、小消息、超大规模
NVLS / CollNet规约直接发生在 NVSwitch 内（NVLink SHARP）交换机帮你做加法，省带宽NVLink 域内、支持 SHARP 的硬件
纠正一个常见误解： "NCCL = Ring AllReduce" 是过时印象。下面深挖区用 Ring 演示是因为它最直观，但真实大规模训练里 Tree 和 NVLS 往往才是实际跑的算法。Ring 是理解基础，不是全部。
新手一句话总结： NCCL = "拓扑探测器" + "把集合通信编译成邻居间传递（Ring / Tree / NVLS 自动选）" + "贴着最快的线零拷贝地搬"。你几乎从不直接调它，框架替你调。
02

集合原语：可交互拆解

"集合通信"= 一组 GPU 一起完成的数据操作。下面有两个视图：① 通用视图只对比通信前 vs 通信后（输入长什么样、输出长什么样），帮你先记住"这个原语到底干了什么事"；② 再往下的深挖视图才逐步拆解中间每一步。先看 ①。每种颜色代表"来自某张卡的数据块"。

先看懂图里那一竖列格子，否则全是天书： 每张 GPU 下面有 4 个上下叠的格子——这是把这张卡持有的数据向量切成 4 段（叫 chunk）。格子里的数字/颜色 = 这一段的数值来自哪张卡（0=GPU0、1=GPU1…），Σ = 已经是"4 张卡对应段求和"的结果。
所以 AllReduce 一开始 GPU0 显示 0 0 0 0，意思是"GPU0 有一个完整向量、切成 4 段、这 4 段当然全是它自己的值"；结束时变成 Σ Σ Σ Σ，意思是"每段都成了所有卡之和"。
⚠️ 这里的 chunk 是算法为了在环上分段流水切的，跟 DDP 的 bucket（梯度分桶）是两回事——bucket 是"把零碎梯度攒成一桶再发"的工程优化，本节不涉及。

关键恒等式： AllReduce = ReduceScatter + AllGather。这不是巧合——正是 Ring AllReduce 的两个阶段，也正是 FSDP 把 AllReduce "拆开用" 的原因。记住它，第 3 节会反复出现。

深挖 · 全部 5 个原语逐步动画

4 张卡（环方向 0→1→2→3→0），每张持有一个切成 4 个 chunk 的向量。选一个原语，用「下一步」逐步观察。

先说清"一步"是什么意思（否则你点了也不知道在看啥）： 这里的每一步 = 所有 GPU 同时各发一个 chunk 给它的右邻居、同时各收一个。不是"GPU0 发完轮到 GPU1"，而是 4 条链路并发跑——这正是环最聪明的地方（没有任何一张卡是热点）。所以 N=4 张卡的 AllReduce 需要 2(N−1)=6 步：前 3 步累加（ReduceScatter），后 3 步传播结果（AllGather）。看图时盯住：绿色描边的格子=这一步正在被发送的 chunk，飞动的小绿点=数据在链路上移动，格子里的 0/1/2/3=数据来自哪张卡，a+b / Σ=已经累加了哪些卡的值。

每个动画都是严格按真实算法推导的状态机，不是随便画的示意：AllReduce / ReduceScatter / AllGather 用 Ring，Broadcast 用二叉树，All-to-All 是逐轮转置。

03·A

DDP — 数据并行

每张卡放整个模型的完整副本，喂不同的数据批次。各自算梯度，然后用 AllReduce 把梯度求平均 → 所有副本走同一步参数更新，保持一致。是最简单、最常用的并行。

▣ DDP 一个训练 step 的时间线关键技巧：梯度分桶，反向边算边通信（overlap）

用到的原语

时机原语
启动：广播初始权重Broadcast
每步反向：求和梯度AllReduce
求和 ≠ 平均，分清归属： NCCL 的 AllReduce 只做 ncclSum（把 N 张卡的梯度相加）。除以 N 求平均是框架/PyTorch 自己做的，不是 NCCL 的事——DDP 通过对梯度做 scale 实现。本页凡说"平均"，底层都是「NCCL 求和 + 框架缩放」两步。
显存代价： 每张卡都存完整模型+梯度+优化器状态。模型放得下单卡才能用 DDP，否则需要 FSDP/TP。

锚定代码 PyTorch

train_ddp.pytorch.nn.parallel
import torch.distributed as dist
from torch.nn.parallel import DistributedDataParallel as DDP

# 后端选 nccl —— 这一行就是在用 NCCL
dist.init_process_group("nccl")
torch.cuda.set_device(local_rank)

model = DDP(model.to(local_rank),
            device_ids=[local_rank])
# ↑ 构造时内部 Broadcast(rank0 权重) 给所有卡

for x, y in loader:
    out = model(x)            # forward
    loss = loss_fn(out, y)
    loss.backward()        # ← 反向钩子触发 AllReduce(梯度)
    optimizer.step()       # 各卡梯度已一致，更新同步
    optimizer.zero_grad()

高亮行：backward() 触发 DDP 注册的 autograd 钩子，梯度凑满一个桶就异步发起一次 ncclAllReduce，和后续梯度计算重叠。

03·B

FSDP — 全分片数据并行

DDP 的痛点是"每卡存整模型"。FSDP（即 ZeRO-3）把参数、梯度、优化器状态都按卡切片：平时每张卡只存自己那 1/N。临用临凑——某层要计算时，AllGather 临时拼出完整权重，用完即扔。这是它如何用 AllReduce = ReduceScatter + AllGather 这条恒等式省显存的。

到底"怎么分到不同卡"？

假设一层有参数向量 W（比如 1024 个数），4 张卡。FSDP 做的事极其朴素：把 W 按下标切成 4 段，卡 i 只在显存里留第 i 段。梯度 G、优化器状态 Opt（Adam 的动量/方差）同样按这个切法各留 1/4。三样东西的"归属"完全对齐——谁拥有第 i 段参数，就负责第 i 段的梯度和优化器状态。这就是"分到不同卡"的全部含义：不是把不同的层放到不同卡（那是 Pipeline 并行），而是把同一层的同一个张量横切开。

▣ 静息布局：一层的 W / G / Opt 如何切到 4 张卡平时（不计算时）每卡显存里只有彩色那 1/4，灰色部分根本不存在

切片的三个级别：ZeRO-1 / 2 / 3

FSDP 背后是 ZeRO（Zero Redundancy Optimizer）的思想，切的东西越多，省的显存越多，通信也越多，分三级递进：

级别切谁每卡还冗余存什么省显存
ZeRO-1只切 优化器状态 Opt完整 W + 完整 G少（但 Adam 的 Opt 通常最占地，先切它性价比高）
ZeRO-2切 Opt + 梯度 G完整 W中
ZeRO-3 = FSDP切 Opt + G + 参数 W几乎没有冗余（≈ DDP 显存 / N）最多
注意别和 Ring AllReduce 的"步骤"混了。 ZeRO-1/2/3 是"切多少种东西"的三个配置档位，不是一次通信的三个先后阶段。PyTorch 的 FSDP 默认就是 ZeRO-3（三样全切）；下面那张"某一层时间线"演的也是 ZeRO-3。如果只想要 ZeRO-1/2，可以用 ShardingStrategy 调（如 SHARD_GRAD_OP ≈ ZeRO-2）。
"用完即扔"扔的是什么？ 扔的是临时 AllGather 拼出来的完整 W——它只在算这一层的几毫秒里存在，算完立刻释放，所以峰值显存只多出"当前这一层的完整权重"，而不是整个模型。参数本体（那 1/4）一直留着，不扔。
▣ ZeRO-3 运行时：处理"某一层"的时间线静息时只存 1/N（见上图），真要算这层时才临时凑出完整权重

用到的原语

时机原语
前向：取本层完整权重AllGather
反向：再取一次权重AllGather
反向：分发并求和梯度ReduceScatter
记忆锚： FSDP = 把 DDP 的 AllReduce 拆成 ReduceScatter（反向）+ 平时把权重 AllGather 回来。省显存，换来约 1.5× 通信。

锚定代码 PyTorch FSDP

train_fsdp.pytorch.distributed.fsdp
from torch.distributed.fsdp import \
     FullyShardedDataParallel as FSDP

dist.init_process_group("nccl")

# 按 Transformer block 自动切分包装
model = FSDP(
    model,
    auto_wrap_policy=transformer_wrap,
    device_id=local_rank,
)   # 参数此刻已被切成 1/N 存放

for x, y in loader:
    out = model(x)        # 每层 fwd 前 AllGather 权重
    loss = loss_fn(out, y)
    loss.backward()    # AllGather 权重 + ReduceScatter 梯度
    optimizer.step()   # 只更新本卡持有的那 1/N 分片

两个高亮处都会触发 NCCL：forward 里每层进出做 ncclAllGather；backward 里 AllGather 取权重、再 ncclReduceScatter 收梯度分片。

03·C

TP — 张量并行

DDP/FSDP 切的是"数据/参数的拥有权"，单层计算仍在一张卡上完成。TP 不一样：它把单个矩阵乘法本身横/竖切开，分给多张卡并行算，再用集合通信把结果拼回。用于单层都大到放不下、或想压低单层延迟的超大模型（Megatron-LM 的做法）。

▣ Megatron MLP：列切 → 行切，一次前向只需 1 个 AllReduceY = Dropout( GeLU(X·A) · B )

用到的原语

时机原语
前向：合并行切结果AllReduce
反向：合并输入梯度AllReduce
有时切序列/词表AllGather / All-to-All
注意通信位置： TP 的 AllReduce 在每一层内部、前向和反向各一次，频率远高于 DDP（每步一次）。所以 TP 几乎只在单节点 NVLink 内用——跨机带宽撑不住。

锚定代码 Megatron 风格

tp_mlp.pycolumn + row parallel
class ParallelMLP(nn.Module):
    def __init__(self):
        # A: 按列切，输出维度分到各卡
        self.fc1 = ColumnParallelLinear(h, 4*h)
        # B: 按行切，配 reduce 合并
        self.fc2 = RowParallelLinear(4*h, h)

    def forward(self, x):
        x = gelu(self.fc1(x))   # 本地，无通信
        x = self.fc2(x)         # row-parallel 末尾→AllReduce
        return x

# RowParallelLinear 内部本质：
def forward(self, x):
    y = x @ self.weight_shard      # 各卡算部分和
    dist.all_reduce(y)            # ← ncclAllReduce 合并
    return y

高亮行：all_reduce(y) 把两张卡的"部分和"加成完整输出。这正是第 2 节 AllReduce 动画演示的操作，只不过这里的"数据"是一个矩阵乘的中间结果。

实战调试速查

真正写分布式训练时，90% 的 NCCL 问题靠这几个环境变量和报错模式定位。

报错对照表

症状多半是
NCCL timeout / watchdog某个 rank 没到（挂了/分支不一致），所有卡死等
unhandled system error网卡/IB 没配好，或 IFNAME 选错
训练能跑但慢回退到了 socket/PCIe；看 DEBUG=INFO 是否走了 NVLink/IB
启动即 hangmaster 地址/端口不通，或进程组数对不上
性能基准： 用 nccl-tests 的 all_reduce_perf -b 8 -e 2G -g 8 测真实带宽，对照硬件理论值，能快速判断"是不是 NCCL 没走对线"。
规则：集合调用必须对齐。 所有 rank 必须以相同顺序、相同 shape 调用同一个集合原语。代码里出现 if rank==0: all_reduce(...) 这种分支，就是 timeout 的经典原因——0 号在通信，别人在等它做别的事。
